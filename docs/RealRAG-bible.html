1. TL;DR

RealPage's codebases are huge, span decades, cross multiple repos. Engineers waste hours tracing simple questions. RealRAG turns any RealPage codebase into a living knowledge base: paste a TFS URL, the system reads the code, writes its own documentation, stores it in a vector database, and answers plain-English questions in under 2 seconds with file-and-line citations. Updates itself on every commit. Works across products with strict isolation. Open-source defaults at every layer.

The product in one line: Paste a TFS URL → ask questions → get cited answers in seconds → docs stay fresh forever.

What it does

Reads any RealPage codebase, generates layered documentation, indexes it into a vector database, and answers questions with citations.

How it stays accurate

Every claim cites file:line. Verifier drops unsupported claims. Refusal mode says "I don't know" instead of hallucinating.

How it stays fast

Pre-computes everything at index time. Caches at every layer. Sub-2-second answers on 95% of questions.

How it stays fresh Phase 2

Manual "Refresh repo" button in the admin panel for the initial release. Webhook-driven auto-refresh is designed and deferred to Phase 2.

2. The Problem (Plain English)

RealPage has many products (RMS, OneSite, ILM, etc.). Each product is built from multiple code repositories. Together it's millions of lines of code, written across decades, by people who have mostly left the company.

When anyone needs to understand how something works — for example, "How is the Revenue Trend Widget data populated?" — they have to:

  1. Search the UI repo (rm-web) for the widget
  2. Find the API call name it sends
  3. Search the API repo (ys) for that API name
  4. Read the SQL it runs
  5. Identify the database table
  6. Search the backend repo (po) for code that writes that table
  7. Read the ETL job, the Java logic, the upstream calculations…

This takes hours per question. Onboarding a new engineer takes months. Existing tools all fail:

  • grep / IDE search — only sees one repo at a time, no semantic understanding
  • Confluence / SharePoint — goes stale within weeks of being written
  • ChatGPT / Copilot scoped to one repo — can't trace across repos, hallucinates without citations

The business knowledge is locked inside the code, and there's no living, trusted, cross-repo source of truth.

3. The Vision

A platform where any RealPage engineer pastes a TFS repo URL and gets back a living, queryable knowledge base in minutes. The system reads the code, writes its own documentation, indexes it, and answers questions with citations and worked examples in under 2 seconds. Updates itself on every commit. Works across all products. Available to engineers, the PME team, and business owners.

4. The Core Loop

  1. Onboard — engineer pastes a TFS URL + PAT in the config screen.
  2. Ingest — system clones the repo, parses it, generates layered documentation, indexes everything into Postgres + FAISS.
  3. Pick a product — user selects RMS / OneSite / ILM. All subsequent queries are scoped to that product.
  4. Query — engineer asks a plain-English question; system returns a cited answer with worked examples in < 2 s.
  5. Fall back — if the docs aren't enough, the system reads the live cloned source via an agentic walk; the new finding feeds back into the index.
  6. Stay fresh — for the initial release, a "Refresh repo" button in the admin panel triggers re-indexing on demand. Phase 2: webhook-driven auto-refresh on every commit (design captured in §7.3 but not implemented now).

Why this shape

  • Postgres holds structure (entities, relationships, facts). It's queryable and joinable.
  • FAISS holds similarity (vectors). It's fast for "find similar."
  • The two are linked by chunk_id. Every retrieval joins them.
  • Per-product isolation at every layer (DB rows tagged with product_id; FAISS files separated by product folder).
  • Cloned source on disk for the rare case where the indexed view isn't enough — the agent reads the live file directly.

6. All Questions Asked — Locked Answers

Will the code itself be stored in the vector database?
Yes — in two tiers. FAISS holds the embeddings (vectors). Postgres holds the actual code text. They're linked by chunk_id. When Claude answers, it reads the code text retrieved from Postgres after FAISS finds the right chunks.
When new code is committed, will the documentation update?
Yes. A TFS webhook fires on every commit. The system runs git diff, finds the changed files, re-parses them, and re-indexes only entities whose body actually changed (entity-content hashing). A typical PR is fully re-indexed in 30–60 seconds for under $1.
How will documentation be generated as detailed as possible, so nothing is missed?
Six strategies, layered:
  1. Atomic units — every function, class, SQL query, XML entry, config key, table column, test gets its own record.
  2. Schema-constrained generation — LLM must fill every required field; cannot skip.
  3. Parser-first extraction — facts (signatures, types, table refs) come from parsers, never the LLM.
  4. Six perspectives per entity — structural, behavioral, business, edge cases, worked example, cross-references.
  5. Three-pass hierarchical generation — entity (cheap LLM) → module (mid) → cross-module narrative (heavy).
  6. Coverage matrix + auto-fill — gaps are tracked; nightly job fills the largest gaps first.
How will retrieval be very fast?
Pre-warm FAISS in RAM at startup; per-repo isolation so we search only relevant data; parallel retrieval (vector + keyword + symbol-exact + graph) via async; multi-level caching (query, embedding, prompt); streaming so the user sees the first word in < 500 ms; tier routing so 60% of queries take the cheapest path. Result: median answer in ~700 ms, p95 in ~1.8 s.
How will answers be very accurate?
Twelve layered defenses:
  1. Hybrid retrieval (vector + keyword + symbol-exact + graph)
  2. HyDE for vague questions
  3. Cross-encoder re-ranking
  4. Domain glossary expansion
  5. Citation-strict prompt + structured output
  6. Refusal mode (says "I don't know" instead of hallucinating)
  7. Verifier pass: citation validity + symbol grounding + faithfulness check
  8. Self-RAG iteration when confidence is low
  9. Knowledge-graph triples for structural questions (no LLM, 100% accurate)
  10. Tests as ground-truth behavior
  11. Eval framework with golden test set
  12. Feedback loop — failed questions feed self-healing
How will cross-repo dependencies be tracked? Example: Revenue Trend Widget data comes from ys repo but is built in po repo.
Per-repo extraction lands in one shared Postgres schema. Custom extractors pull legacy patterns (TS string literals like query: 'X', XML query catalog entries, ETL XML services, SQL FROM/JOIN refs). A merge pass stitches edges by exact string match into a cross_repo_edges table. At query time, a recursive Postgres CTE walks the graph in < 50 ms, retrieves docs at each step, and the LLM narrates the chain with citations.
If I don't tell you the example, will you understand the cross-repo connection?
Partially — honest answer. Three strategies fill the gap:
  • Hand-curated YAML pattern catalog — one senior engineer writes the cross-repo patterns once (one day of work); reused forever. Best accuracy.
  • LLM-assisted pattern discovery at onboarding — LLM reads samples, proposes patterns, human accepts. Catches the obvious ones automatically.
  • Co-occurrence statistical discovery — nightly job finds new patterns from token frequencies. ~85% accurate.
We use all three.
What's the strategy for chunking, indexing, embedding?
Chunking: AST-aware via tree-sitter; one chunk per atomic unit; never split mid-function. Multi-chunk-type indices (code, doc, generated, examples, tests, schema, config).
Indexing: Postgres for structure + FAISS for similarity; per-repo per-product isolation; atomic upsert across both stores in a single transaction; entity-content hashing to skip unchanged code; index versioning.
Embedding: BAAI/bge-large-en-v1.5 (open source, 1024-dim) or OpenAI text-embedding-3-large (3072-dim); same model at index and query time; batched (100 chunks per call); persistent cache; multi-vector per entity (signature, body, summary embedded separately).
Will you create a database per product?
No — one Postgres database with product_id as a column on every table. FAISS files are physically separated by product folder. Defense-in-depth ACL: API route → query → (optional) Postgres row-level security. Switch to per-product physical databases only if compliance demands it later.
After cloning, will the code be stored on disk?
Yes — partial+shallow clones (~10% of full size) kept long-term under storage/repos/<product>/<repo>/<branch>/. Reasons: incremental git fetch on commits, agentic source-walk fallback, git blame for "why" questions, re-parse without re-clone. Encrypted at rest; per-product permissions; deleted only when a repo is removed from the config screen.
When a user clicks a product name, will queries be scoped to that product?
Yes — product context is the user's primary scope. Once "in" RMS, every retrieval, every document, every example is scoped to RMS. Glossary changes per product. URL becomes /api/v1/products/{product_id}/…. Switching products is one click. Cross-product queries require explicit opt-in.
For every business rule, will an example be available?
Yes — every rule stored carries at least one verified worked example as a mandatory field of the same record. Pydantic schema rejects rules without examples. The system literally cannot store a rule without a paired example. At query time, rule + example are retrieved together; no on-the-fly synthesis needed.
Doesn't pairing examples increase documentation cost?
Yes — about 30–50% one-time index cost increase, ~10% storage. But:
  • Per-query cost goes down (no on-the-fly synthesis)
  • Pre-stored examples pay for themselves on the first ~30 queries that hit them
  • Bounded by tiered richness: trivial entities get no example, important ones get full examples
Net: cheaper to operate, faster to answer, more consistent across users.
What about the database schema for RMS / other products?
Not blocking. The system extracts table/column references from code regardless. When you provide schema dumps (pg_dump --schema-only dropped into db/external_schemas/), accuracy on SQL/table claims jumps from ~85% to ~98%, and column-level docs become available.
Do we need agentic RAG?
Yes, but tiered — not always-on. Three modes routed by intent:
  • Tier 1 — Naive RAG (60% of queries): one retrieval, one answer, < 1 s, $0.001
  • Tier 2 — Self-RAG (25%): LLM critiques retrieval, re-retrieves if weak
  • Tier 3 — Agentic RAG (12%): multi-step tool use for cross-repo / multi-hop questions
  • Tier 4 — Live source-walk (3%): reads cloned files when the index is insufficient
What's the admin / config screen?
A web UI for: managing repos (paste TFS URL, trigger re-index, remove); cost dashboards per product/team; tuning thresholds (FAISS score floor, top-K, tier rules); reviewing LLM-proposed cross-repo patterns; user/ACL management; audit log search; coverage matrix per repo; failed-question queue.
What embedding model is used in the existing AI-RMS today?
OpenAI text-embedding-3-large (3072 dimensions). Pluggable via EMBEDDING_MODEL env var. For RealRAG production: open-source default BAAI/bge-large-en-v1.5 (1024 dims, MIT license).

7.5 — Config Screen Inputs (TFS link + Repo Overview) CRITICAL

The config screen captures more than just a URL. The repo overview entered by the engineer onboarding the repo is consumed at indexing time and at query time — it is not optional metadata. Without it, the system has no human-curated grounding for the repo's purpose, owners, or critical entry points.

The overview is load-bearing. The same TFS URL pasted by two different engineers should produce two different indexing experiences if the overviews differ. Treat overview fields as first-class input, not decoration.

10. Concepts Glossary — Plain English

One line per concept. What it is. Why it matters.

ConceptWhat it isWhy we use it
RAG (Retrieval-Augmented Generation)Looking up relevant chunks of code/docs and giving them to the LLM as context, instead of relying on the LLM's training data.The whole product is built on it. Stops hallucinations because answers are grounded in real code.
Naive RAGOne look-up, one LLM answer. The simplest form.Fast and cheap; works for 60% of questions.
Self-RAGThe LLM checks if the look-up was good, and if not, asks for more before answering.Catches ambiguous questions where the first look-up missed something.
Agentic RAGThe LLM is given tools (search, graph-walk, read-file) and plans a multi-step investigation.Needed for cross-repo / multi-hop questions like "trace from UI to database."
HyDEMake up a fake answer first, embed that fake answer, then look up real chunks similar to it.Vague questions don't match real chunks well; a hypothetical answer matches them better.
EmbeddingA list of 1024 (or 3072) numbers that represents the meaning of a chunk of text.Lets us find "similar" chunks fast; the basis of vector search.
Vector databaseA database optimized for "find vectors similar to this one." We use FAISS.Sub-30 ms similarity search across millions of chunks.
FAISSFacebook's open-source library for fast similarity search.Industry standard; battle-tested; sub-30 ms top-K.
ChunkA small piece of code or doc — usually one function, one class, one SQL query.The unit we embed and retrieve. Smaller = more precise; bigger = more context.
AST-aware chunkingSplitting code at function/class boundaries (using a parser) instead of arbitrary line counts.Preserves logic; embeddings make sense. Cutting mid-function destroys meaning.
tree-sitterAn open-source parser library that turns code into an Abstract Syntax Tree for any language.The de-facto standard. Handles Java, TypeScript, SQL, Python, etc.
Hybrid retrievalRunning vector search + keyword search + symbol exact-match all in parallel and merging results.Each catches what the others miss. Single biggest accuracy lever.
Re-rankingTaking the top 50 results and using a smarter (slower) model to score them more carefully, keeping the best 5.Lifts accuracy by 15% for ~80 ms latency cost.
Cross-encoderA model that scores a (question, chunk) pair together, more accurately than vector similarity alone.The standard re-ranker (e.g., BAAI/bge-reranker-v2-m3).
Citation groundingEvery claim in the answer must reference a specific file:line.Wrong answers become visibly wrong (the citation doesn't actually say that). Builds trust.
Refusal modeThe system says "I don't have evidence" instead of guessing.Better silence than hallucination. Users learn to trust it.
Verifier passAfter the LLM generates an answer, a second pass checks every claim against the source.Catches hallucinations that slipped past the prompt. Drops unsupported sentences.
Schema-constrained generationForcing the LLM to fill a typed JSON schema (every field required).Guarantees nothing is skipped. The LLM literally cannot return an incomplete doc.
Few-shot promptingIncluding 2-3 perfect examples in the prompt before asking the LLM to produce the next one.Locks in voice and structure. LLMs match style much better than they follow instructions.
Prompt cachingAnthropic caches the system prompt + retrieved chunks for 5 minutes; repeat calls pay only for the new tokens.~60% token savings on follow-up turns of the same conversation.
Tier routingPicking a cheap (Haiku), medium (Sonnet), or expensive (Opus) LLM based on question complexity.30-100× cost savings vs always using the heaviest model.
Streaming (SSE)Sending the answer to the user word-by-word as the LLM generates it.First token in < 500 ms even when the full answer takes 3 s.
Knowledge graph triplesStoring facts as (subject, predicate, object) like (functionX, calls, functionY).Some questions ("what writes to table X?") are answered by SQL on triples — provably 100% accurate, no LLM.
Cross-repo edgesConnections between entities in different repos (rm-web sends query name → ys defines that name).Lets us trace UI → API → SQL → ETL → table across the whole RealPage stack.
Pattern catalogA YAML file describing how cross-repo connections work in this organization.One day of work by a senior engineer; reused for every repo onboarded.
Worked exampleA concrete scenario with real numbers showing how a business rule plays out.Stored alongside every rule. Users get a concrete answer for "give me an example."
Domain glossaryList of business terms (propcode, RRG, useAdjustedRent) injected into every prompt.Forces consistent vocabulary; prevents the LLM from inventing synonyms.
Diátaxis frameworkDocumentation has four types: Reference, Explanation, How-to, Tutorial. Each answers different questions.Generate all four per entity = covers any question.
Map-reduce summarizationChunk a big file, summarize each chunk, then summarize the summaries.Defeats "lost-in-the-middle." Big files don't break the LLM.
Entity-content hashA fingerprint of a function's body. We store the hash and skip re-indexing if the hash didn't change.Whitespace edits and comment fixes cost zero LLM calls. Real changes cost only what's truly different.
Atomic upsertUpdating Postgres + FAISS together in a single transaction. Both succeed or both roll back.Never leaves a half-indexed broken state.
Index versioningTagging the index with (embedding_model, schema_version, parser_version). Bumping a version triggers a full rebuild.Lets us swap the embedding model without breaking anything.
Idempotent webhookSafe to call the same webhook twice with the same payload — second call is a no-op.TFS sometimes retries; we don't double-index.
Dead-letter queue (DLQ)Failed jobs go to a holding area for manual review.Errors don't disappear silently.
Coverage matrixA table showing per-entity which doc types are filled and which are missing.Surfaces gaps. Nightly job auto-fills the largest gaps first.
Self-healingFailed user questions go to a queue. The next nightly run regenerates docs for those entities at deeper quality.The system gets measurably better at the questions your team actually asks.
RAGASAn open-source evaluation framework for RAG systems.Measures faithfulness, context relevance, etc. Catches regressions when prompts change.
Prompt injectionAn attacker hides instructions inside a chunk so the LLM follows them ("ignore previous instructions, do X").We sandbox retrieved content and structure prompts to resist this.
PAT (Personal Access Token)Authenticated string that proves you have permission to read a TFS repo.How we authenticate to TFS for cloning. Stored in a vault, never on disk.
Partial + shallow clonegit clone with --filter=blob:none --depth 50 — pulls only recent commits and skips most file blobs.Cuts disk usage to ~10% of full clone. Still enough for tree-sitter to parse.
Product contextThe RMS / OneSite / ILM the user is currently scoped to. All queries auto-filtered to that product's repos.Enforces isolation. RMS questions never accidentally retrieve OneSite chunks.
