🧠 How RealRAG Was Built

A deep dive into the implementation — the problem, the approach, the decisions, and the lessons learned building a RAG system on top of a large enterprise codebase.

RAG System 5 Repos Indexed Multi-Tier Chat Hackathon Project
The Problem

Why was this built?

🚨 Core Pain Point

The RMS codebase spans 5+ repositories, millions of lines of Java/TypeScript/SQL, and years of business logic. When a developer, QA engineer, or business analyst asks "How does renewal pricing work?" — there is no fast answer. They have to dig through code, ask a senior dev, or look through Confluence pages that may be stale.

The core insight was that the codebase IS the documentation — it's just written in a language (Java/SQL) that most people on the team can't quickly read. RealRAG translates it into plain business English on demand.

"The goal was not to build a chatbot. It was to make the codebase accessible to every person on the team — dev, QA, BA, and product — without requiring them to read code."

The specific challenges this solves:

  • New team members spend weeks ramping up — RealRAG reduces this to hours
  • Business analysts can't verify if the code matches the spec without a developer's help
  • Cross-repo questions (e.g., "how does rm-web call ys-reports?") require tracing 3-4 repos manually
  • SQL stored procedures contain critical business logic that very few people understand
The Approach

How the solution was designed

The core idea was a multi-stage pipeline that transforms raw source code into a searchable knowledge base of business-readable documentation. Rather than asking an LLM to read raw code at query time (slow, expensive, unreliable), the expensive work is done once during indexing.

📖 Index-Time Intelligence

Claude reads and understands each code entity ONCE during indexing, writes human-readable docs. At query time, the LLM only needs to read those pre-written docs — not the raw code.

🔍 Hybrid Retrieval

No single search method works for all questions. Vector search finds semantically similar content; keyword search finds exact terms; symbol match finds class names; KG finds relationships.

📊 Auto-Escalation Tiers

Most questions can be answered with a fast retrieval pass. Only genuinely hard questions escalate to deeper, more expensive modes. This keeps average response time low.

💬 Business Language First

The system prompt strictly forbids code references in answers. Users should never see "see ReportProcessorServiceImpl.java:237" — they should see a plain English explanation.

Implementation

Building the Indexing Pipeline

The pipeline was the first thing built because without indexed data, there is nothing to chat about. It runs once per repo and produces everything the chat system needs.

1

Cloning from TFS

The first challenge was connecting to RealPage's TFS. Implemented lib/clone.py with PAT authentication and shallow clone support (shallow_50 by default) to avoid downloading gigabytes of git history. Clone depth is configurable per repo.

2

Language-Aware Parsing with tree-sitter

The RMS codebase is primarily Java + TypeScript + SQL. Used tree-sitter for Java and TypeScript — it provides concrete syntax trees that correctly handle complex patterns like Spring annotations, generics, and nested classes. Each parser produces standardized ParsedEntity objects, making the rest of the pipeline language-agnostic.

3

6-Perspective Documentation Generation

The key innovation was the 6-perspective doc schema in lib/doc_gen.py. For each code entity (class, method, SQL procedure), Claude is prompted to write: (1) structural overview, (2) behavioral description, (3) business purpose, (4) edge cases, (5) worked example, (6) cross-references. This produces docs that are useful for many different question types — not just "what does this do" but also "when would this fail?" and "how does this connect to X?"

4

Three-Pass Doc Generation

Pass 1 (doc_gen.py) writes per-entity docs. Pass 2 (doc_gen_pass2.py) rolls these up into per-file module summaries — better for "what does this file do?" questions. Pass 3 (doc_gen_pass3.py) writes cross-repo narratives using the knowledge graph edges — essential for cross-repo questions like "how does rm-web send data to ys?"

5

Embedding with On-Disk Cache

lib/embedder.py sends chunks to OpenAI's text-embedding-3-large. The critical optimization was an on-disk SHA1 cache — if the same text has been embedded before, it returns immediately without an API call. This made re-indexing after partial failures fast and free.

6

Per-Repo FAISS Indexes

Chose FAISS (Facebook AI Similarity Search) over managed vector DBs for two reasons: (1) it runs entirely locally — no external service needed, (2) per-repo indexes allow product-scoped search so a question about YieldStar doesn't pull in PO results. Each repo gets its own .faiss file under storage/faiss/.

Retrieval Design

Building the Hybrid Search Engine

Pure vector search is not enough. The core insight from building lib/retrieval.py was that different question types need different search strategies:

Vector Search (FAISS)

Best for: "How does renewal work?" — finds semantically similar content even when exact words don't match. "Lease expiry" finds "renewal" content.

Keyword Search (Postgres FTS)

Best for: "What is YieldStar?" — exact term matches. Business terminology like product names and feature names are reliably found by keyword, not always by vector.

Symbol Exact-Match

Best for: "What does ReportProcessorService do?" — when a developer mentions a specific class name, go directly to it rather than doing a fuzzy search.

Knowledge Graph (SQL Triples)

Best for: "What writes to the lease table?" — structural questions about data flow. The KG was built by extracting entity relationships during parsing into facts stored as SQL triples.

All five methods run in parallel using Python asyncio, and results are merged and re-ranked. This means the total latency is the slowest single method, not the sum of all methods.

Chat Design

Building the 4-Tier Chat Router

The main insight in lib/chat.py was that most questions don't need an expensive search. The auto-escalation design means 80% of questions are answered cheaply in Tier 1, and only the hard cases pay the cost of Tier 2/3.

T1

Tier 1 — Fast (~5s)

Standard retrieval: top-5 hits, Claude Haiku. The faithfulness checker validates the answer: Does it look uncertain? Does it name things not in the retrieved context (hallucination signal)? If strong → done. If weak → escalate.

T2

Tier 2 — Deep (~10s)

Same retrieval but with top-20 hits and broader search radius. The faithfulness check was the key innovation — rather than always doing deep search, it only escalates when the answer looks genuinely weak.

T3

Tier 3 — Agent (~30s)

lib/agent.py gives Claude a set of tools (search, symbol lookup, KG query, grep) and lets it plan its own multi-step investigation. This is what handles cross-repo questions that need multiple search rounds to piece together an answer.

T4

Tier 4 — Live Agents (~60s+)

lib/claude_runner.py launches the Claude Code agents that already exist in each repo's .claude/agents/ folder — headlessly, without any UI. This lets RealRAG run the same deep investigation agents that developers use, but triggered from a chat question.

Lessons Learned

Challenges & How They Were Solved

❌ Problem: Code vocabulary ≠ Business vocabulary

"Renewal" in business becomes "lease_renewal_processor" in code. Pure vector search on code chunks returned irrelevant results for business-worded questions.

✅ Solution: Business doc arm + Glossary

Built a separate FAISS index for business-language HTML docs (RealRAG-bible, scope, glossary). Also injected the RMS glossary into every LLM prompt so term mapping is consistent.

❌ Problem: LLM hallucinations

Early versions confidently cited Java classes that didn't exist or described features that were wrong. The retrieval was finding the right area but the LLM was filling gaps with invented details.

✅ Solution: Faithfulness checker + Auto-escalation

Added answer_has_fabrication() and context_covers_subjects() validators. If the answer mentions subjects not in the retrieved context, it auto-escalates to deeper search before returning.

❌ Problem: Re-embedding cost on reruns

Re-running the pipeline after a bug fix was expensive — re-embedding all chunks cost money and time even when 99% of the code hadn't changed.

✅ Solution: SHA1-keyed disk cache

The embedder caches vectors keyed by SHA1 of the text. Re-running the pipeline after small changes only re-embeds the chunks that actually changed — everything else is free.

❌ Problem: 8192 token limit on embeddings

Some generated docs were longer than 8192 tokens, causing OpenAI API errors during the embed step.

✅ Solution: Smart truncation in embedder.py

Added tiktoken-based truncation in lib/embedder.py that trims to 8000 tokens while preserving complete sentences. The embedding captures the most important content.

❌ Problem: Chat history lost on page reload

Streamlit re-runs the entire script on every interaction, losing all session state including conversation history.

✅ Solution: Persistent sessions in PostgreSQL

Built lib/sessions_repo.py to save every message to the database. On app load, the latest session is restored. Conversations survive restarts, page reloads, and server crashes.

❌ Problem: Answers leaking code references

Early LLM answers said things like "see RenewalPricingEngine.java line 247" — useful for developers, completely useless for business users.

✅ Solution: Strict system prompt + Citation separation

The system prompt explicitly forbids class names, file paths, SQL table names, and line numbers in the answer text. Citations are shown separately in a collapsible "Sources" section.

Architecture

Key Design Decisions

🏗️ Index-time vs Query-time LLM

Chose to run LLM at index time (doc gen) rather than passing raw code to LLM at query time. Slower to set up, but dramatically faster queries and much lower per-question cost. One entity doc is generated once and reused for thousands of questions.

🗄️ Embedded PostgreSQL

Used pgserver (embedded Postgres) instead of a cloud database or SQLite. Gives full SQL power with zero infrastructure — no server to manage, no connection strings to configure. Team members just install the Python package.

🔑 Product-Scoped Indexes

Each product (RMS) has its own FAISS index. Searching for YieldStar features doesn't surface results from the PO codebase. Product scoping is set per-question in the UI sidebar.

📦 Per-Repo FAISS Files

Kept FAISS indexes per-repo rather than one merged index. This allows re-indexing one repo without touching others, makes debugging easier, and keeps individual index sizes manageable.

🤖 Reuse Existing Agents

Rather than building new agents for Tier-4, headlessly invoked the Claude Code agents already living in each repo's .claude/agents/ folders. Zero duplication — the same agents devs use in VS Code are reused for chat.

🚫 No LLM in Parsing

Parsing, chunking, and knowledge-graph construction are fully deterministic — no LLM calls. This keeps the pipeline fast, deterministic, and cheap. LLM is only invoked for doc generation (expensive, runs once) and answering (cheap, runs per question).
