2. Data Flow Map

How information moves through the system. The AI must be able to trace any of these paths end-to-end.

READ PATH (user views a widget) ─────────────────────────────────────────────────────────────── rm-web component "I want the seasonal-adj widget data" │ │ HTTP POST { query: 'getSeasonalAdjustmentWidgetData' } ▼ ys controller / service "lookup that named query" │ │ NamedQueries.getNamedQuery('...') → yscore_queries.xml entry ▼ yscore_queries.xml SQL FROM seasonalforecast SF JOIN ... │ │ executes against client DB ▼ client DB (Postgres) reads from seasonalforecast table │ │ rows returned ▼ ys → JSON → rm-web chart renders WRITE PATH (nightly ETL fills the data) ─────────────────────────────────────────────────────────────── po EtlToPOExecutor (TOPOETL) "load tonight's seasonalforecast" │ ▼ etl2posql.xml <service name="SeasonalForecast"> FROM ... │ ▼ po Forecasting / Rates pipeline computes adjustmentratio │ ▼ client DB INSERT INTO seasonalforecast REPORT PATH (user runs a JASPER report) ─────────────────────────────────────────────────────────────── rm-web "View Report" → ys-reports resolves report by name │ ▼ ys-reports JRXML + parameters compiles and runs │ ▼ client DB / ysmaster / ysmasteranalytics runs JASPER's SQL │ ▼ PDF / HTML / CSV stream → rm-web streamed back CONFIG PATH (admin onboarding ETL) ─────────────────────────────────────────────────────────────── ys-admin UI → ys-admin services writes config rows │ ▼ poconfig DB properties · servers · ETL recipes │ ▼ po reads poconfig at startup knows which client DBs to hit

  • Widget name + screen + route path
  • The query name string sent to ys (the bridge to ys query catalogs)
  • Data shape returned + how it's rendered
  • User actions on the widget (filters, drill-downs)
  • i18n labels and warning messages (e.g. INSUFFICIENT_SUBJECT_DATA_RENEWAL_SEASONAL_FORECAST_ADJUSTMENT)

4.3 Indexing strategy for ys

  • Each <entry key="X"> in every catalog is its own atomic entity (deeply documented)
  • Each Java service method that calls NamedQueries.getNamedQuery("X") is cross-linked to the matching XML entry
  • SQL bodies parsed with sqlglot to extract FROM/JOIN/INSERT/UPDATE tables → cross-link to po ETL writers
  • REST controllers indexed with their endpoint paths; cross-link to rm-web call sites

4.4 What the AI must answer

  • "What does query X return?"
  • "What tables does query X touch?"
  • "Which UI component triggers query X?"
  • "Which po ETL fills the table that query X reads?"
  • "What's the difference between yscore and airm queries?"

6.4 Indexing strategy for po (deepest of any repo)

  • Every Java method/class indexed at L3 (full doc + worked example) — not just L1
  • Every etl2posql.xml service entry indexed as its own atomic entity
  • Every sql.xml named query indexed
  • Statement-level annotation for the four heart ETLs (TOYSMaster, TOPOETL, Forecasting, Rates Generation)
  • Capture R-script integration points (forecasting calls R via shell)
  • Cross-link every ETL → tables it writes → ys queries that read those tables → rm-web widgets
  • Capture every config flag that affects ETL behavior (poconfig keys, properties)
Heart of the application. The user explicitly said: "TOYSMaster, TOPOETL, Forecasting, Rates Generation are very most important; store as minute information as it can." These four get the deepest treatment — statement-level annotations, multiple worked examples per business rule, full git-history context.

8.1 Indexing strategy for databases

  • One chunk per table (with all columns, types, constraints, FKs, indexes, comments)
  • One chunk per view, stored procedure, trigger
  • Cross-link every code reference (Java, XML, SQL) to its table
  • Build FK graph; enable graph traversal for "what tables join to X?"
  • Drift detection: code references column that doesn't exist → admin alert

10.1 The pattern catalog

Each pattern above gets a YAML entry. One day of work by a senior engineer to write the catalog; reused for every repo onboarded.

# patterns/realpage_cross_repo.yaml
patterns:
  - name: "ys-query-dispatch"
    from: { repo: rm-web, extractor: ts-string-near-key, key: "query|queryName" }
    to:   { repo: ys, extractor: xml-entry-key, files: "*_queries.xml" }
    edge: dispatches_query
    confidence: 1.0

  - name: "etl-table-write"
    from: { repo: po, extractor: etl-xml-write, files: "etl2posql.xml" }
    to:   { table_ref: any }
    edge: writes_table
    confidence: 1.0

  - name: "named-query-usage"
    from: { extractor: regex, pattern: 'NamedQueries\.getNamedQuery\("([^"]+)"\)' }
    to:   { extractor: xml-entry-key, files: "**/sql.xml" }
    edge: uses_named_query
    confidence: 1.0

  - name: "jasper-report-tables"
    from: { repo: ys-reports, extractor: jrxml-query-string }
    to:   { table_ref: any }
    edge: reads_table
    confidence: 1.0

  - name: "ui-launches-report"
    from: { repo: rm-web, extractor: report-launch-call }
    to:   { repo: ys-reports, extractor: jrxml-name }
    edge: launches_report
    confidence: 0.95

  - name: "poconfig-drives-etl"
    from: { repo: po, extractor: config-key-read }
    to:   { schema: poconfig, extractor: column-or-table }
    edge: reads_config
    confidence: 1.0

13.3 Operational commitments

  • Freshness: docs no older than the last commit on the watched branch (webhook-driven).
  • Citations: every claim cites file:line; refusal mode for unsupported claims.
  • Per-product isolation: users in RMS context see only RMS chunks.
  • Audit log: every question + answer + citations logged for compliance.
  • Self-healing: failed questions queue for nightly deeper-doc-gen.
  • Cost: ~$300 one-time per repo to onboard; sub-cent per query on the warm path.
