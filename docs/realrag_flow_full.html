C. Runtime step details

↑ flow

R1 — Render chat & capture input

The Chat tab draws the conversation and reads your typed question plus the sidebar state: which product to search, the Deep search toggle, the Agent mode toggle, and the New chat button. This is the single entry point for every question.
codefrontend/streamlit_app.py_render_chat_tab()
↑ flow

R2 — Resume session & load history

Reuses the latest saved conversation (or creates one), loads prior messages so the chat survives reloads/restarts, and immediately records your new question. This is what makes the chat persistent and multi-turn.
codelib/sessions_repo.pylatest_session()create_session()load_messages()save_message() · datasessionsmessages
↑ flow

R3 — Mode router (Agent? / Cache?)

Two early exits before the normal path: (1) if Agent mode is ON, the question goes straight to the multi-step agent (RA). (2) if the answer cache is on AND there's no prior context, an exact repeat returns instantly. The cache is currently disabled during development, so this almost always falls through to the fast path.
codefrontend/streamlit_app.py (L804 agent · L820 cache) · lib/query_cache.pycache_get()
↑ flow

R4 — Build conversation context

Folds the last ~4 turns into a context block and keeps the previous user turn, so follow-ups like "give an example of that" resolve correctly and the retrieval search is aimed using the real subject.
codelib/chat.pyprepare_answer() (builds the history block)
↑ flow

R5 — Understand intent

Figures out the kind of question (definition / calculation / list / how-it-works), how hard it is (which model tier), and the key terms to search. Fast path uses a no-LLM heuristic; Deep search uses the LLM classifier, which also rewrites/expands the query (HyDE). Also flags "exact" and "what all / list all" phrasing.
codelib/chat.py_fast_intent()_is_calc_question()wants_exact()_is_enumeration() · lib/intent.pyclassify()
↑ flow

R6 — Embed the question

Turns the question (plus the previous turn) into a meaning-vector via OpenAI, so the app can find material by meaning, not just matching words. Results are cached on disk and oversized text is truncated to fit the model limit.
codelib/embedder.pyembed_texts()_truncate()_load_cached() · datastorage/embeddings_cache/*.npy
↑ flow

R7 — Knowledge-graph shortcut

For fact-style structural questions ("what writes to / reads from table X"), build a deterministic answer directly from the recorded fact triples — exact and complete, no fuzzy search needed.
codelib/chat.py_kg_answer_for() · lib/kg.pywrites_to_table()reads_from_table()find_entity_by_name() · datafacts
↑ flow

R8 — Retrieve: 5 arms in parallel

Five searches run at once and their candidates are pooled, so nothing slips through: (1) meaning-search on code, (2) meaning-search on generated docs, (3) keyword full-text, (4) exact-symbol match (pulls the entity's full doc), (5) curated business-docs. A light cosine gate (_VECTOR_MIN_COS) drops true vector noise.
codelib/retrieval.pyretrieve()vector_arm_code()vector_arm_generated()fts_arm()symbol_arm()business_arm() · lib/faiss_store.pysearch() · lib/business_docs.pysearch_business_docs()
↑ flow

R9 — Fuse & rank

Merges the five lists into one ranked shortlist. Pieces several arms agree on rise (Reciprocal Rank Fusion); then nudges: plain-English docs get a boost, calculation questions bias toward pricing/SQL (UI down), and known critical entry-points get lifted. Either code or docs can win — the boosts are nudges, not rules.
codelib/retrieval.py_fuse_and_rank()_calc_weight()_critical_boost()_doc_rank()_dedup_key()
↑ flow

R10 — LLM re-rank (Deep search only)

A second, smarter pass: the AI reads the top candidates and reorders them by which truly answers the question — fixing cases where the math ranking was fooled by surface signals. Costs an extra AI call, so it only runs in Deep search (or auto-escalation).
codelib/retrieval.py_llm_rerank() → lib/llm.pycall_json()
↑ flow

R11 — No matches → code-search fallback

If every arm comes back empty, the app abandons the pre-built index and greps the actual cloned source files for your keywords, then answers from those snippets — or honestly says it doesn't know rather than guessing. A safety net for under-documented corners.
codelib/chat.py_code_fallback_answer() → lib/code_search.pysearch_code() · datastorage/repos/* (cloned files)
↑ flow

R12 — Assemble context (special cases)

Two adjustments before writing: Enumeration ("what all / list all") pulls EVERY entry of the dominant file so the list is complete, not truncated. Exact ("formula / which table / columns") switches to a prompt that is allowed to name precise artifacts, relaxing the usual no-code-references rule.
codelib/chat.py_is_enumeration()_file_full_listing()wants_exact()_system_blocks_stream_exact()
↑ flow

R13 — Build prompt + pick model tier

Assembles the final instruction package: history + KG facts + glossary definitions + retrieved context + your question. Then routes to a model by difficulty — T1 Haiku (simple), T2 Sonnet (mid), T3 Opus (complex reasoning).
codelib/chat.py_build_prompt()_system_blocks_stream()_TIER_TO_MODEL · lib/glossary.pyglossary_block()
↑ flow

R14 — Generate the answer (streamed)

The chosen model writes the reply in plain business language with no code references in the prose, and the text streams onto the screen token-by-token so it feels fast and responsive.
codelib/chat.pystream_answer() → lib/llm.pystream_text() · frontend/streamlit_app.py_stream_to()
↑ flow

R15 — Self-check (is the answer weak?)

After writing, the app grades its own answer. It's flagged weak if ANY of: it sounds unsure/refuses, it names something not in the retrieved sources (fabrication), or the question's subject isn't actually covered (retrieval drifted). A solid answer ends here.
codelib/chat.pyanswer_looks_unsure()answer_has_fabrication()context_covers_subjects() · frontend/streamlit_app.py_weak() (L838)
↑ flow

R16 — Escalation Step 1: Deep search

If weak (and the Deep toggle was OFF), the app re-runs the question in Deep search (smarter intent + LLM re-rank + stronger model) and replaces the answer in place. This fires when the fast answer is weak OR was a "final"-mode (non-streamed) answer.
codefrontend/streamlit_app.py L845-854 · lib/chat.pyprepare_answer(deep_search=True)
↑ flow

R17 — Escalation Step 2: Agent loop

Exact gate: the agent runs only if the Deep-search answer is still weak AND it was not a "final"-mode deterministic answer. It investigates step-by-step, then its result replaces the answer. If the Deep answer was good, the agent never runs. (Note: when you turn Deep search ON manually, this auto-ladder is skipped entirely.)
codefrontend/streamlit_app.py L856-863 · lib/agent.pyrun_agent()
↑ flow

RA — The Agent loop (detail)

The model plans up to ~6 tool calls; each result is fed back and it decides the next move, then writes a grounded business answer. Tools: search_knowledge_base (full hybrid retrieve), find_symbol, what_writes_to_table / what_reads_table, trace_cross_repo (follow UI→API→SQL→ETL→table), grep_source, get_documentation.
codelib/agent.pyrun_agent()_dispatch()_entity_doc() · reuses lib/retrieval.pylib/kg.pylib/code_search.py
↑ flow

R18 — Render answer + sources

Shows the streamed answer and a collapsible Sources list (the citations behind it), so you can see what the answer was grounded in.
codefrontend/streamlit_app.py_render_sources()
↑ flow

R19 — Persist + cache + scroll

Saves the question + answer + sources so the conversation survives reloads; writes to the answer cache only if the answer is grounded & confident (cache currently off); auto-scrolls the view to the new answer.
codelib/sessions_repo.pysave_message() · lib/query_cache.pycache_put() · frontend/streamlit_app.pycomponents.html()

D. Offline step details

↑ flow

O1 — Register repo

Capture repo metadata: source URL, product, repo role, priority, critical entry points, languages. RMS repos can be added from presets.
codefrontend/streamlit_app.py · lib/repos_repo.py · lib/presets.py · lib/models.py · dataproductsrepos
↑ flow

O2 — Clone

Git-clones the repository to local storage so parsers and the code-search fallback can read the real files.
codelib/clone.pyclone_repo() · datastorage/repos/*
↑ flow

O3 — Walk & classify files

Lists every file, detects its language, and skips binaries, generated, vendored, and test files so only meaningful source is indexed.
codelib/walker.pywalk() · datarepo_files
↑ flow

O4 — Parse each file (per-language dispatch)

The chunker routes every file to the right parser and extracts atomic units (classes, methods, queries, services, functions) plus facts (reads/writes table). XML uses structural detection (count <service> vs <entry>) so query catalogs and ETL catalogs aren't confused; non-DDL SQL files yield SQL functions/procedures.
codelib/chunker.pychunk_repo()_parse_dispatch() · lib/parser_java.pyparse_java_file() · lib/parser_typescript.py · lib/parser_xml.pyparse_xml_file() · lib/parser_sql.pyparse_sql_file()extract_sql_facts() · lib/parser_ddl.py · lib/parser_jrxml.py · lib/parser_markdown.py
↑ flow

O5 — Persist pieces + facts

Saves entities, code/doc chunks, and fact triples to Postgres. A body-hash skips unchanged pieces on re-runs, so re-indexing is incremental.
codelib/chunker.py_persist_entities_recursive()_persist_chunks_and_facts() · lib/db.py · dataentitiescode_chunksdoc_chunksfacts
↑ flow

O6 — Doc-gen Pass 1 (entity docs, Haiku)

For each meaningful piece, the LLM writes a structured doc with 6 perspectives (structural, behavioral, business, edge-cases, worked example, cross-refs), grounded in the glossary and checked by a verifier. These plain-English docs are what most answers are built from.
codelib/doc_gen.pydoc_gen_repo()_list_entities_for_scope()agenerate_entity_doc() · lib/glossary.py · lib/llm.py · datagenerated_docs
↑ flow

O7 — Cross-repo edges

Applies a hand-curated pattern catalog to link UI → API → SQL → ETL → table across repos — deterministic, no LLM. This powers cross-repo tracing.
codelib/cross_repo.pydiscover_edges() · data/patterns_realpage.yaml · datacross_repo_edges
↑ flow

O8 — Pass 2 / L4 enrich / Pass 3

Pass 2 (Sonnet) rolls entity docs into file/module narratives; L4 (PO only) adds extra worked examples, statement-level annotations, and git-history context; Pass 3 (Opus) narrates end-to-end cross-module workflows.
codelib/doc_gen_pass2.pydoc_gen_pass2_repo() · lib/doc_gen_l4_enrich.py + lib/git_history.py · lib/doc_gen_pass3.py · datagenerated_docsexamples
↑ flow

O9 — Embed & build indexes

Embeds all code chunks and generated docs (batched, cached, 8192-token truncation), writes per-repo FAISS indexes (code.* and generated.*), and builds the curated business-doc index. These are the indexes the runtime search arms query.
codelib/embed_pipeline.pyembed_repo() · lib/embedder.pyembed_texts() · lib/faiss_store.pywrite_index() · lib/business_docs.pybuild_business_index() · datastorage/faiss/*
↑ flow

O10 — Coverage + self-healing

Tracks how much of each repo has docs and logs questions that failed, so gaps can be filled later.
codelib/coverage.py · dataentity_coveragefailed_questions
Two important runtime facts (verified in code): (1) Auto-escalation (fast→Deep→Agent) only runs when the Deep search toggle is OFF; turning it ON answers once in Deep mode without auto-escalating. (2) The answer cache is intentionally disabled (_CACHE_ON=False) during development so pipeline fixes aren't masked by stale answers.
